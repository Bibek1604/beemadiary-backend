## How to Apply the Migration

### Option 1: Using Prisma CLI (Recommended)

This is the easiest method if your database is running and network access is available.

```bash
cd backend

# Standard migration command
npx prisma migrate deploy

# Or if you need to generate the Prisma client too:
npx prisma migrate dev --name add_notes_table
```

### Option 2: Manual SQL Execution with psql

If the Prisma CLI has issues, you can apply the migration directly using PostgreSQL CLI:

```bash
# Using psql directly
psql $DATABASE_URL -f src/prisma/migrations/20260524000000_add_notes_table/migration.sql

# Or connect interactively and paste the SQL:
psql postgresql://postgres:adminbibek@localhost:5432/TestManagement

# Then paste the contents of:
# src/prisma/migrations/20260524000000_add_notes_table/migration.sql
```

### Option 3: Using the Provided Node Script

A Node.js helper script is available at `run-migration.js`. Make sure your database is running:

```bash
cd backend
npm install pg --save-dev  # if not already installed
node run-migration.js
```

## Migration SQL Details

The migration creates:

1. **NoteTag Enum** - PostgreSQL enum with values: GENERAL, IMPORTANT, FOLLOW_UP, TODO
2. **Notes Table** - Table with:
   - `id` (TEXT, Primary Key) - UUID generated by app
   - `title` (VARCHAR(500), nullable) - Note title, auto-generated from content if not provided
   - `content` (TEXT, NOT NULL) - Note content
   - `tag` (NoteTag, default GENERAL) - Category tag
   - `agent_id` (UUID, NOT NULL) - Reference to agents table
   - `created_at` (TIMESTAMP, default now) - Creation timestamp
   - `updated_at` (TIMESTAMP, auto-update) - Last update timestamp
   - `deleted_at` (TIMESTAMP, nullable) - Soft delete marker
3. **Indexes** - On agent_id, tag, and created_at for query performance
4. **Foreign Key** - agent_id references agents.id with CASCADE delete

## Verification Steps

After running the migration, verify it was successful:

### 1. Check the table exists
```bash
psql postgresql://postgres:adminbibek@localhost:5432/TestManagement -c "\dt notes"
```

Expected output should show the notes table.

### 2. Check the enum exists
```bash
psql postgresql://postgres:adminbibek@localhost:5432/TestManagement -c "\dT+ \"NoteTag\""
```

Expected output should show the NoteTag enum with GENERAL, IMPORTANT, FOLLOW_UP, TODO values.

### 3. Verify indexes
```bash
psql postgresql://postgres:adminbibek@localhost:5432/TestManagement -c "\di notes*"
```

Expected output should show three indexes: notes_agent_id_idx, notes_tag_idx, notes_created_at_idx.

## Testing the API

Once migration is complete, test the Notes endpoints:

### 1. Create a Note
```bash
curl -X POST http://localhost:3000/api/personal-notes \
  -H "Authorization: Bearer YOUR_JWT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "content": "This is my first note",
    "title": "My Note",

  }'
```

## Troubleshooting

### Migration fails with "NoteTag already exists"
This means the enum was already created. You can safely skip this step or drop and recreate the database.

### "relation 'notes' already exists"
The notes table was already created. Verify it has all the correct columns and indexes. If you need to reset:
```bash
# Backup first!
DROP TABLE IF EXISTS notes CASCADE;
```

### Foreign key constraint error
Ensure the `agents` table exists and has data. The notes table references it.

### Permission denied
Ensure your PostgreSQL user has CREATE permission on the schema. Check with:
```bash
psql postgresql://postgres:adminbibek@localhost:5432/TestManagement -c "SELECT current_user;"
```

## Additional Notes

- **Soft Deletes**: Notes are soft-deleted (marked with `deleted_at` timestamp) not permanently removed
- **Ownership**: Only agents can view/edit their own notes (enforced in API routes)
- **Auto-generated Titles**: If no title is provided, the first 60 characters of content become the title
- **Search**: Full-text search on title and content (case-insensitive)
- **Admin Override**: Admins can permanently delete any note using the `/permanent` endpoint
