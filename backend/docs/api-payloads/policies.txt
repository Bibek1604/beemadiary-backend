ENDPOINT
Route: /api/policy/create
Method: POST
Authentication Required: Yes
Roles Allowed: Any authenticated agent
Description: Creates a new policy for an existing client.

Headers:
- Authorization: Bearer <token>
- Content-Type: application/json

Request Body Example:
{
  "client_id": "client-id",
  "plan_name": "Endowment Plan",
  "plan_no": "14",
  "policy_term": "20 years",
  "sum_assured": 1000000,
  "ab_pwb": "AB-001",
  "doc": "2026-05-25",
  "maturity_time": "2046-05-25",
  "premium_amount": 25000,
  "discount_scheme": "5% Corporate Discount",
  "premium_due_date": "2026-12-31",
  "bank_account": "1234567890",
  "branch": "Kathmandu Main Branch",
  "status": "ACTIVE"
}

Validation Rules:
- client_id: required
- plan_name: required, max 100 characters
- plan_no: required, max 50 characters
- premium_amount: required, positive number
- sum_assured: optional but must be positive if supplied
- status: optional and must be one of ACTIVE, INACTIVE, PENDING, LAPSED, EXPIRED
- maturity_time: parsed as a date when supplied

Success Response Example:
{
  "status": true,
  "message": "Policy created successfully",
  "data": {
    "id": "policy-id",
    "data": {
      "id": "policy-id",
      "policy_number": "LIC-POL-1716546834000-ABC12",
      "plan_name": "Endowment Plan",
      "plan_no": "14",
      "premium_amount": 25000,
      "status": "ACTIVE"
    }
  },
  "code": 201
}

Possible Error Responses:
- 400 validation error
- 401 missing token
- 403 unauthorized to create policy for this client
- 404 client not found
- 500 unexpected failure

MongoDB Collections Affected:
- policy
- client

Business Logic Notes:
- policy_number is generated automatically.
- The route accepts a bank_account and branch but not the full bank-details payload.

ENDPOINT
Route: /api/policy/:policyId
Method: PUT
Authentication Required: Yes
Roles Allowed: Owner agent or admin
Description: Updates the selected policy fields.

Headers:
- Authorization: Bearer <token>
- Content-Type: application/json

Request Body Example:
{
  "plan_name": "Updated Plan",
  "plan_no": "15",
  "policy_term": "25 years",
  "sum_assured": 1200000,
  "ab_pwb": "AB-002",
  "doc": "2026-06-01",
  "maturity_time": "2047-06-01",
  "premium_amount": 30000,
  "discount_scheme": "10% Annual Discount",
  "premium_due_date": "2027-01-01",
  "bank_account": "1234567890",
  "branch": "Lalitpur",
  "premium_paid": 5000,
  "status": "ACTIVE"
}

Validation Rules:
- only allowed fields are updated
- premium_amount must be positive
- sum_assured and premium_paid must be non-negative numbers
- status must be ACTIVE, INACTIVE, PENDING, LAPSED, or EXPIRED
- maturity_time must be a valid date if supplied

Success Response Example:
{
  "status": true,
  "message": "Policy updated successfully",
  "data": {
    "id": "policy-id",
    "policy_number": "LIC-POL-2026-001"
  },
  "code": 200
}

Possible Error Responses:
- 400 validation error or no valid fields to update
- 401 missing token
- 403 unauthorized to update this policy
- 404 policy not found
- 500 failed to update policy

MongoDB Collections Affected:
- policy

Business Logic Notes:
- The route performs a soft update and keeps the latest policy linked to the client.

{}

}

Possible Error Responses:
- 401 missing token
- 500 failed to sync lapsed policies

MongoDB Collections Affected:
- policy

Business Logic Notes:
- This endpoint is the automated status migration path for aging policies.
