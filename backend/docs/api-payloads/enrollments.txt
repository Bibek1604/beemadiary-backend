Enrollment image handling rule:
- profile_picture is a single file field and stores exactly one profile image for the client.
- supporting images are sent as an array of files, using supporting_images or images.
- supporting images are stored as an array on the client record in the images field.
- supporting documents are not used by the active enrollment route; that terminology belongs to clientDocuments routes, not client enrollment.

==================================================
ENDPOINT
Route: /api/client/enroll
Method: POST
Authentication Required: Yes
Roles Allowed: Any authenticated agent
Description: Creates a client enrollment record with personal profile, optional policy details, bank details, a single profile picture, and an array of supporting images.

Required enrollment inputs:
- One of full_name or the pair first_name + last_name
- email
- phone
- address

Optional inputs:
- secondary_phone
- dob
- age
- nominee_name
- relation_with_nominee
- profession
- member_group
- gender
- reason_for_insurance
- all policy_details fields
- all bank_details fields
- profile_picture
- images / supporting_images

Headers:
- Authorization: Bearer <token>
- Content-Type: multipart/form-data

Multipart File Fields:
- profile_picture: single image file
- images: array of image files
- supporting_images: array of image files
- image_labels / supporting_images_labels / imageLabels: array of labels matching the image order

Validation Rules:
- full_name is required unless first_name and last_name are provided together
- email required, valid, max 255 characters
- phone required, exactly 10 digits
- address required
- nominee_name required in this implementation
- relation_with_nominee required in this implementation
- gender required and must be male, female, or other
- secondary_phone, when supplied, must be 10 digits
- dob, when supplied, must be a valid date and not in the future
- age, when supplied, must be between 0 and 120
- bank_account must be digits only and max 20 digits
- premium_due_date and date_of_commencement must be valid dates when supplied
- policy_status normalizes to ACTIVE, INACTIVE, PENDING, LAPSED, or EXPIRED
- premium_status only accepts DUE, PAID, or PARTIAL

Possible Error Responses:
- 400 validation error, duplicate email/phone, or failure to generate a unique client id
- 401 missing token
- 500 policy creation failure or unexpected failure

MongoDB Collections Affected:
- client
- policy
- company
- agent

Business Logic Notes:
- The client record is created first, then the policy record is created in a best-effort sequence.
- If the policy create fails, the newly created client is rolled back.
- Supporting images are sent from the frontend as an array, not a single document field.
- The single profile picture is separate and should be sent once using profile_picture.

Validation Rules:
- first_name, last_name, email, phone, address, nominee_name, relation_with_nominee, gender are required
- email must be valid
- phone and secondary_phone must contain 9-15 digits
- dob, if present, must be valid and not in the future
- age, if present, must be between 18 and 100
