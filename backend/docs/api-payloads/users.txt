ENDPOINT
Route: /api/users/me
Method: PUT
Authentication Required: Yes
Roles Allowed: AGENT
Description: Updates the current logged-in agent profile and optionally uploads a new profile picture.

Headers:
- Authorization: Bearer <token>
- Content-Type: multipart/form-data

Request Body Example:
{
  "full_name": "Test Agent",
  "email": "agent@test.com",
  "phone_number": "9999999999",
  "lic_agent_code": "LIC-123",
  "branch_division": "Kathmandu",
  "qualification": "Bachelor",
  "position_designation": "Senior Agent",
  "short_bio": "Experienced insurance agent"
}

Multipart File Fields:
- profile_picture: single image file

Validation Rules:
- full_name: required, non-empty string
- email: required, valid email
- phone_number: required, numeric digits only
- lic_agent_code: optional string
- branch_division: optional string
- qualification: optional string
- position_designation: optional string
- short_bio: optional, maximum 500 characters
- profile_picture: optional image file

Success Response Example:
{
  "status": true,
  "message": "Profile updated successfully",
  "data": {
    "id": "uuid",
    "full_name": "Test Agent",
    "email": "agent@test.com",

    "branch_division": "Kathmandu",
    "qualification": "Bachelor",
    "position_designation": "Senior Agent",
    "short_bio": "Experienced insurance agent",
    "profile_picture": "https://..."
  },
  "code": 200
}

Possible Error Responses:
- 400 validation error or invalid image
- 401 missing or invalid token
- 403 role is not AGENT
- 404 profile not found
- 500 unexpected failure

MongoDB Collections Affected:
- agent

Business Logic Notes:
- The controller delegates to the user service and stores the uploaded file through Cloudinary/local storage.

ENDPOINT
Route: /api/admin/users
Method: POST
Authentication Required: Yes
Roles Allowed: SUPER_ADMIN, ADMIN
Description: Creates an admin, agent, or client record depending on the supplied role.

Headers:
- Authorization: Bearer <token>
- Content-Type: application/json

Request Body Example:
{
  "role": "AGENT",
  "username": "agent007",
  "first_name": "Test",
  "last_name": "Agent",
  "email": "agent2@example.com",
  "phone_number": "9876543210",
  "password": "ChangeMe123!",
  "is_active": true,
  "company": "company-id"
}

Role-specific examples:
- ADMIN: username, email, phone_number, password, is_active
- AGENT: username or first_name/last_name, email, phone_number, password, is_active, company
- CLIENT: first_name, last_name, email, phone_number, address, password, agent_id optional

Validation Rules:
- password: required
- role defaults to AGENT
- unique constraints are enforced by the database

Success Response Example:
{
  "status": true,
  "message": "Agent created successfully",
  "data": {
    "id": "uuid",
    "email": "agent2@example.com",
    "role": "AGENT",
    "is_active": true
  },
  "code": 201
}

Possible Error Responses:
- 400 validation error, duplicate email/phone, or invalid reference
- 401 unauthorized
- 500 could not create user

MongoDB Collections Affected:
- admin
- agent
- client
- company

Business Logic Notes:
- Passwords are hashed before persistence.
- Client creation auto-selects the first available agent if none is supplied in some flows.

ENDPOINT
Route: /api/admin/users/:id
Method: PATCH
Authentication Required: Yes
Roles Allowed: SUPER_ADMIN, ADMIN
Description: Updates whichever user collection owns the provided id.

Headers:
- Authorization: Bearer <token>
- Content-Type: application/json

Request Body Example:
{
  "username": "agent008",
  "first_name": "Updated",
  "last_name": "Name",
  "email": "agent8@example.com",
  "phone_number": "9800000000",
  "address": "Kathmandu",
  "is_active": false,
  "company": "company-id",
  "password": "NewPassword123!"
}

Validation Rules:
- only supplied fields are updated
- password is optional but must be strong if provided

Success Response Example:
{
  "status": true,
  "message": "User updated successfully",
  "data": {
    "id": "uuid",
    "email": "agent8@example.com",
    "role": "AGENT",
    "is_active": false
  },
  "code": 200
}

Possible Error Responses:
- 400 validation error or duplicate field
- 401 unauthorized
- 404 user not found
- 500 could not update user

MongoDB Collections Affected:
- admin
- agent
- client

Business Logic Notes:
- This route performs a soft role-aware update based on the id lookup order.

ENDPOINT
Route: /api/admin/users/bulk_activate_deactivate
Method: POST
Authentication Required: Yes
Roles Allowed: SUPER_ADMIN, ADMIN
Description: Bulk activates or deactivates a set of users across all user collections.

Headers:
- Authorization: Bearer <token>
- Content-Type: application/json

Request Body Example:
{
  "user_ids": ["uuid-1", "uuid-2"],
  "is_active": true
}

Validation Rules:
- user_ids: non-empty array of ids
- is_active: boolean

Success Response Example:
{
  "status": true,
  "message": "Users updated successfully",
  "data": {
    "ids": ["uuid-1", "uuid-2"],
    "is_active": true
  },
  "code": 200
}

Possible Error Responses:
- 400 validation error
- 401 unauthorized
- 500 unexpected failure

MongoDB Collections Affected:
- admin
- agent
- client

Business Logic Notes:
- Applies the same active/inactive flag across all user collections.
